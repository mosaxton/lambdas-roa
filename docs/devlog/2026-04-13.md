## What This Repo Is

`roa-lambdas` is the Python side of the Right of Access system. The web app (`app.rightofaccess`) handles everything up to and including the OAuth flow — at the end of that flow it puts a message in an SQS queue that says "go get this claimant's insurance data." This repo is what picks that message up.

It is a separate GitHub repo from the web app by design: different runtime (Python on AWS Lambda vs TypeScript on Amplify), different deployment cadence, different IAM permissions. See `docs/superpowers/plans/roa-lambdas-standalone-project.md` in the web app repo for the full reasoning.

This is the first session's worth of work on this repo. Plan reference: `app.rightofaccess/docs/superpowers/plans/roa-lambdas-standalone-project.md`.

## Decisions Made Before Coding

These were open questions in the plan, resolved before any code was written:

| Decision | Choice | Why |
|---|---|---|
| Dependency management | pip-tools | Simpler than Poetry for a Lambda-only repo; `pip-compile` produces a locked, reproducible `requirements.txt` |
| Shared code packaging | Lambda Layer | Encryption code must not exist in N separate copies that can drift |
| Local Postgres | docker-compose | Simple, no daemon needed; `make db-up` spins it up |
| Postgres in CI | Testcontainers (inside pytest) | No docker-compose dependency in GitHub Actions runners |
| Prisma schema source | Git submodule at `vendor/app.rightofaccess` | Full clone (not sparse); CI contract-drift check reads `vendor/app.rightofaccess/prisma/schema.prisma` directly |
| Token refresh for MVP | Inline in `fhir_processor` only | Proactive `token_refresh` Lambda deferred to post-MVP |
| NPPES cache TTL | 30 days prod / 7 days dev | Controlled by `NPPES_CACHE_TTL_HOURS` env var |
| `fhir_processor` concurrency | Start at 3 | Conservative vs. NPPES rate limit (~100 req/min); raise after measuring real throughput |

**`samconfig.toml`** — SAM CLI configuration. Points `dev` at the root `infra/template.yaml`, sets the S3 deployment bucket, region, and confirms changeset automatically for dev.

**`vendor/app.rightofaccess`** — Git submodule pinned to the web app repo. The contract-drift script reads `vendor/app.rightofaccess/prisma/schema.prisma` to verify that every column the Python code references still exists with the same name.

**`events/`** — Sample event payloads for local testing:

### Shared Layer (`shared/`)

The shared layer is packaged as a Lambda Layer. Every function imports from it. Nothing in here should be Lambda-specific — these are pure Python utilities.

**`shared/encryption.py`** — Complete.

The Python twin of `app.rightofaccess/lib/encryption.ts`. Uses AES-256-GCM via the `cryptography` library. The byte layout is non-negotiable and must stay in sync with the TypeScript side forever: `[IV (12 bytes)][authTag (16 bytes)][ciphertext (N bytes)]`. This layout is what allows a token encrypted by the Next.js app to be decrypted by the Lambda.

Key loading: reads `ENCRYPTION_KEY` from the environment, validates it's a 64-character hex string (32 bytes), converts hex to raw bytes. Raises a `ValueError` immediately if the key is missing or malformed. Never logs the key value.

Two functions: `encrypt(plaintext: str) -> bytes` and `decrypt(data: bytes) -> str`. Each call to `encrypt()` generates a fresh IV.

**`shared/db.py`** — Complete.

The only place in the entire roa-lambdas codebase where raw SQL lives. Lambda handlers call helpers from this file; they never write `cursor.execute()` directly. Same rule as `lib/db/` in the web app.

Connection lifecycle: a module-level `psycopg2` connection is created on the first call and reused across warm Lambda invocations (Lambda containers stay alive between calls for performance). If the connection drops between invocations (TCP timeout, RDS restart), it reconnects automatically. The `get_connection()` context manager commits on clean exit and rolls back on any exception.

All helpers implemented:

| Helper | What it does |
|---|---|
| `get_payer_token(conn, case_id, payer_slug)` | Fetch the full encrypted token row |
| `update_payer_token(conn, token_id, ...)` | Update token + expiry after a refresh |
| `list_expiring_tokens(conn, window_minutes)` | Find tokens expiring within N minutes |
| `get_case(conn, case_id)` | Fetch case id, firm_id, status, payer_slug |
| `update_case_status(conn, case_id, status)` | Set case status + updated_at |
| `upsert_eob_raw(conn, case_id, fhir_resource_id, raw_json_enc)` | Store/update encrypted raw EOB |
| `upsert_provider(conn, case_id, npi, ...)` | Insert or update provider row, return UUID |
| `insert_encounter(conn, case_id, provider_id, ...)` | Add an encounter row |
| `insert_prescription(conn, case_id, provider_id, ...)` | Add a prescription row |
| `get_nppes_cache(conn, npi)` | Cached NPPES lookup |
| `upsert_nppes_cache(conn, npi, data)` | Insert or refresh NPPES cache entry |
| `update_payer_health(conn, payer_slug, ...)` | Upsert payer health status + failure counter |
| `insert_audit_log(conn, action, ...)` | Append audit log entry |

### Tests (`shared/tests/`)

**`shared/tests/test_encryption.py`** — 11 tests. Covers: round-trip (empty string, ASCII, Unicode, 10KB), IV uniqueness (encrypting the same string twice produces different bytes), byte layout verification (length = 28 + plaintext bytes), error cases (buffer too short, tampered auth tag, missing env var, invalid key length), and — most critically — the cross-language fixture test.

The cross-language fixture test (`test_cross_language_fixture_decrypt`) reads `shared/tests/fixtures/ts_encrypted.bin` (bytes produced by the Node.js crypto library using a known key and known plaintext) and asserts that the Python `decrypt()` function produces the correct plaintext. If this test fails, the two sides have drifted and no OAuth tokens stored by the web app can be read by the Lambda. This is the first test that should pass before any other Lambda work begins.

**`shared/tests/fixtures/ts_encrypted.bin`** — Binary file committed to the repo. Generated by `make gen-fixtures` (runs `scripts/generate-cross-lang-fixture.ts`). Do not edit manually.

**`shared/tests/fixtures/ts_encrypted_meta.json`** — Metadata for the fixture: the hex encryption key and the expected plaintext. Used by the test to set the correct `ENCRYPTION_KEY` and verify the decrypted output.

**`shared/tests/test_db.py`** — Integration tests for every helper in `shared/db.py`. Each test gets its own psycopg2 connection that is rolled back at the end (via `conftest.py`), so tests are fully isolated without truncating tables between runs.

Coverage includes:
- `get_payer_token` found/not found
- `update_payer_token` updates the right columns
- `list_expiring_tokens` finds expiring tokens, ignores already-expired ones, ignores fresh ones
- `get_case` and `update_case_status`
- `upsert_eob_raw` insert and idempotent update
- `upsert_provider` returns UUID, idempotent on NPI conflict
- `insert_encounter` with JSONB dx_codes and cpt_codes
- `insert_prescription` all fields
- `get_nppes_cache` miss and hit
- `upsert_nppes_cache` idempotent
- `update_payer_health` insert, increment consecutive_failures, reset to 0
- `insert_audit_log` persists all fields correctly

### Infrastructure (`infra/`)

The SAM templates are complete. They define what gets deployed — not the Lambda code itself (which isn't written yet), but the AWS resources that will run it.

**`infra/template.yaml`** — Root stack. Imports all per-function stacks as nested CloudFormation stacks. Also creates the shared IAM execution role (used by all Lambdas: Secrets Manager read, SQS consume, CloudWatch write, VPC ENI) and an SSM Parameter Store entry that publishes the SQS queue URL so the Next.js app can read it without a hardcoded env var.

**`infra/shared-layer.yaml`** — Defines the Lambda Layer that packages `shared/`.

**`infra/fhir-processor.yaml`** — SQS queue (`eob-processing-{env}`), dead letter queue, the `FhirProcessorFunction` Lambda (3 reserved concurrent executions, 300s timeout, 512MB), and CloudWatch alarms (error rate, p99 duration, DLQ depth > 0).

**`infra/payer-health-check.yaml`** — `PayerHealthCheckFunction` (128MB, 60s timeout) + EventBridge rule (rate 5 minutes).

**`infra/token-refresh.yaml`** — `TokenRefreshFunction` (256MB, 3min timeout) + EventBridge rule (rate 15 minutes).

**`infra/dlq-alerter.yaml`** — `DlqAlerterFunction` (128MB, 30s timeout) triggered by the DLQ itself.

**`infra/cold-storage-mover.yaml`** — `ColdStorageMoverFunction` skeleton. EventBridge rule exists but is disabled. Phase 2 — not for MVP.

**`infra/parameters/{dev,staging,prod}.json`** — Per-environment SAM parameters: VPC subnet IDs, security group IDs, Secrets Manager secret names, Blue Button base URL.

## What to Build Next

Following the build order in `docs/superpowers/plans/roa-lambdas-standalone-project.md` §11:

1. **`shared/payer_registry.py`** — Python mirror of the web app's payer registry. Simple dataclass + dict. Required by `fhir_processor` to know the FHIR base URL and token URL for each payer.
2. **`shared/secrets.py`** — AWS Secrets Manager loader, cached per cold start. Reads `roa/database-url`, `roa/encryption-key`, `roa/bb-client-id`, etc.
3. **`shared/logging.py`** — Structured JSON logging with PHI field denylist. Every log line is one JSON object (CloudWatch indexes this).
4. **`functions/fhir_processor/eob_parser.py`** — Pure function, easy to TDD. Converts raw FHIR EOB resources into normalized `{providers, encounters, prescriptions}` dicts. Has the most test fixtures.
5. **`functions/fhir_processor/fhir_client.py`** — Paginated FHIR fetcher with backoff. Follows `Bundle.link[relation=next]`.
6. **`functions/fhir_processor/nppes_resolver.py`** — NPI → provider details, cache-aware.
7. **`functions/fhir_processor/handler.py`** — SQS entry point. Wires together all of the above + inline token refresh.
8. **Deploy to dev and run end-to-end smoke test** with Blue Button sandbox (`BBUser00000 / PW00000!`).
9. **`functions/payer_health_check/handler.py`**
10. **`functions/dlq_alerter/handler.py`**

Read `vendor/app.rightofaccess/wiki/payer-bluebutton.md` before touching the FHIR client or EOB parser — it has the sandbox URLs, test accounts, and Blue Button-specific FHIR quirks.
